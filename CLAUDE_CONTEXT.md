## ⚠️ Known Issues & Gotchas

> Key gotchas Claude must know. Full detail in CLAUDE_CONTEXT_HISTORY.md if needed.

### Auth — CRITICAL
- **`accessToken`** = Google OAuth token (~1hr). Only for Drive/Calendar API. NEVER guard Supabase ops with `!accessToken`.

### Shared-Secret `X-PRVS-Secret` Pattern (Session 50)
- Browser-originated edge-function calls that must bypass JWT verification authenticate instead via a shared secret: HTTP header `X-PRVS-Secret: <PRVS_FUNCTION_SECRET>`.
- Edge functions read `Deno.env.get('PRVS_FUNCTION_SECRET')` and compare against `req.headers.get('x-prvs-secret')`. Mismatch → 401.
- CORS preflight (`OPTIONS`) must include `x-prvs-secret` in `Access-Control-Allow-Headers` — otherwise Chrome blocks the header on the actual POST.
- Client side: `index.html` stores the secret as a top-of-file JS constant `PRVS_FUNCTION_SECRET` and appends the header on 4 fetches: `photo_share`, `parts_ordered`, `parts_eta_update`, `parts_request`. `customer-checkin.html` uses the same pattern for `lead_staff_notify`.
- **Secret rotation:** Update in Supabase project Settings → Edge Functions → Secrets AND in both HTML files' `PRVS_FUNCTION_SECRET` constants. Redeploy affected edge functions.

### `--no-verify-jwt` Deployment Flag (Session 50)
- `send-quote-email` MUST be deployed with `supabase functions deploy send-quote-email --no-verify-jwt`. Without the flag, Supabase requires a valid user JWT on every call — which would break `customer-checkin.html` (unauthenticated public page) and the shared-secret pattern.
- Root cause: ES256 JWT alg required by Supabase's default JWT verifier is incompatible with the `apikey`/anon-key tokens the browser can emit from an unauthenticated page. Shared-secret `X-PRVS-Secret` replaces JWT auth for these paths.
- Same `--no-verify-jwt` flag applies to: `slack-notify` (origin-only auth).
