## Time-Tracking Hardening (Session 52–53)

### Auto-Logout Regime Change (12h cap → M-F 5 PM EOD)
- **Old (Session 52, killed Session 53):** `auto_close_stale_time_logs()` daily at 02:00 UTC capped any open session >12h old at `clock_in + 12h`. Roland found this was inflating labor costs (forgotten clock-outs counted 12h of phantom labor). Killed entirely — cron unscheduled, function dropped.
- **New (Session 53):** `close_open_time_logs_eod()` SECURITY DEFINER function + pg_cron `close-open-time-logs-eod` (job ID 8) at `0 22 * * 1-5` (M-F 22:00 UTC = 5 PM CDT). Closes open sessions with `close_reason = 'auto_eod_5pm'`. Respects `extended_at` (skips sessions extended within the last hour — for future Twilio YES-reply extension). Weekend stragglers stay open until Monday's cron.
- **Retroactive fix:** 9 prior `auto_eod`+12h rows rewritten to `clock_in + 8h` with `close_reason = 'auto_eod_8h_recalc'`.

### GH#33 Phase 1 — Rapid-Fire Clock-In Dupes (Session 53)
DB-level UNIQUE partial index `idx_time_logs_one_open_per_tech_ro ON time_logs (tech_email, ro_id) WHERE clock_out IS NULL` makes a second open clock-in for the same (tech, RO) physically impossible — any INSERT returns 23505 unique_violation. Partial predicate allows legitimate re-clock-in after clock-out.

`checkin.html` v1.30 hardens the client around it: `_clockInPending` re-entry guard, instant button lock (disabled + spinner + opacity 0.6), pre-INSERT SELECT guard, 23505 race-window backstop, offline-queue replay dedupe in `drainQueue()`, Spanish translation for the new toast. Kills both root-cause patterns (3-second double-tap and 8-second offline-replay storm).
